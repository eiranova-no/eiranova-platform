## Bakgrunn

K-DB-001 etablerte komplett database-schema (26 tabeller, RLS, indexes,
seed-data) på dev-Supabase (`hlenyzppjodmkoaugzid`). prod-Supabase
(`wxyfwgxvfhpicmcpknxt`) er fortsatt tomt (0 tabeller).

K-ENV-002 innførte env-guard som krever at `NEXT_PUBLIC_SUPABASE_URL` og
`NEXT_PUBLIC_SUPABASE_ANON_KEY` er satt i begge Vercel-scopes (Preview og
Production). I dag er kun Preview-scope satt for plattform-prosjektene
(eiranova-web, eiranova-nurse, eiranova-admin), noe som fører til
500-feil (`MIDDLEWARE_INVOCATION_FAILED`) på Production-URL-er som
`app.eiranova.no`.

Denne kontrakten lukker gapet ved å:
1. Migrere prod-Supabase til samme schema som dev-Supabase
2. Sette Production env-vars i Vercel som peker på prod-Supabase
3. Verifisere at både Preview og Production fungerer end-to-end mot
   sine respektive Supabase-prosjekter

Etter merge er miljø-oppsettet komplett og videre arbeid kan fokusere
utelukkende på funksjonalitet.

## Mål

Etablere strikt miljø-skille i drift:

- **Preview-deploys** (PR-er, dev-branch): treffer dev-Supabase, viser EnvBadge
- **Production-deploys** (`app/nurse/admin.eiranova.no`): treffer prod-Supabase, ingen EnvBadge

Begge miljøer skal returnere 200/307 på root-URL og kunne rendres uten
runtime-feil.

## In Scope

### Migrasjon av prod-Supabase
- Linke Supabase CLI til prod-prosjektet (`wxyfwgxvfhpicmcpknxt`)
- Kjøre alle 8 migrasjoner fra `supabase/migrations/` med `supabase db push`
- Verifisere at alle 26 tabeller eksisterer i prod
- Verifisere at RLS-policies er aktivert
- IKKE migrere data — prod skal starte tomt (ingen brukerdata)

### Vercel Production env-config
For alle tre plattform-prosjekter (`eiranova-web`, `eiranova-nurse`,
`eiranova-admin`):

| Variabel | Verdi | Scope |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | `https://wxyfwgxvfhpicmcpknxt.supabase.co` | Production |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `<prod anon key>` | Production |
| `NEXT_PUBLIC_APP_ENV` | `production` | Production |
| `NEXT_PUBLIC_DATA_SOURCE` | `prod` | Production |

### Redeploy og verifisering
- Trigge Production-deploy for alle tre prosjekter
- Verifisere at Production-URL returnerer 200/307
- Verifisere at Preview fortsatt fungerer (regresjon)
- Verifisere at EnvBadge er skjult i Production, synlig i Preview

## Out of Scope

- Endring av Vercel Production Branch (forblir `dev` til K-LAUNCH-001)

- Endring av custom domener
- Avvikling av `marketing` og `oppstart` (K-LAUNCH-001)

- Backup-strategi for prod-Supabase (egen oppgave senere)

## Manual Steps (Richard)

Følgende krever Richard fordi de gjelder Supabase-passord, GitHub secrets
eller Vercel-konfig som Cursor ikke har tilgang til.

### Steg 1: Verifiser/installer Supabase CLI
```bash
supabase --version
# Hvis ikke installert:
brew install supabase/tap/supabase
```

### Steg 2: Hent prod database-passord
- Supabase → eiranova-prod → Settings → Database → Reset database password
- Lagre i 1Password
- IKKE commit passord til repo

### Steg 3: Link og push migrasjoner
```bash
cd ~/Code/EiraNova  # eller hvor repoet ligger
supabase link --project-ref wxyfwgxvfhpicmcpknxt
# (oppgi prod-passord når CLI spør)
supabase db push
```

### Steg 4: Verifiser i Supabase Dashboard
- prod-Supabase → Database → Tables
- Skal vise samme 26 tabeller som dev
- prod-Supabase → Authentication → Policies
- Skal vise RLS-policies aktivert

### Steg 5: Hent prod anon key
- prod-Supabase → Settings → API → anon public → Copy

### Steg 6: Sett Production env-vars i Vercel
For hvert av prosjektene `eiranova-web`, `eiranova-nurse`, `eiranova-admin`:

- Settings → Environment Variables
- Legg til 4 variabler med scope **Production** (se tabell over)

### Steg 7: Redeploy alle tre prosjekter
- Vercel → prosjekt → Deployments → siste Production-deploy → `...` →
  Redeploy → uncheck "Use existing Build Cache"

### Steg 8: Verifiser
```bash
curl -I https://app.eiranova.no/
curl -I https://nurse.eiranova.no/
curl -I https://admin.eiranova.no/
```

Forventet: 200 eller 307. Ikke 500.

Browser-test:
- Åpne hver Production-URL — skal laste uten EnvBadge
- Åpne en Preview-URL fra siste PR — skal laste med EnvBadge

## Risks

- **Migrasjons-feil:** Hvis en migrasjon feiler på prod (annerledes
  startpunkt enn dev), må feil tabeller dropes manuelt og migrasjonen
  kjøres på nytt. Prod er tomt, så lav risiko.
- **Rate limiting:** Supabase Pro har generøse grenser, men `supabase db
  push` kan ta noen minutter på 8 migrasjoner.
- **Anon key-eksponering:** Anon key er offentlig (bakes inn i klient-
  bundle), så ingen sikkerhetsrisiko ved å sette i Vercel UI. Service-role
  key derimot — skal ALDRI settes i NEXT_PUBLIC_*-variabler.
- **EnvBadge-logikk:** Per K-ENV-001/002 vises badge når
  `APP_ENV !== 'production'`. Production setter den til `production`,
  så badge skjules automatisk.
- **Cache i Vercel:** Husk uncheck "Use existing Build Cache" ved
  redeploy. NEXT_PUBLIC_*-vars bakes inn ved build.
